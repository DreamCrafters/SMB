# Инструкции для AI-агентов

## Обязательный контекст

- Для задач в этом проекте использовать локальный skill `smb-monitor-project`, если он доступен.
- Перед изменением кода, архитектуры, документации или порядка задач читать `implementation_plan.md`.
- Перед изменением модели данных, аккаунтов, доступов, источников данных или серверных контрактов читать `docs/data-architecture.md`.
- Перед изменением backend, БД, миграций, API endpoints или запуска серверной части читать `docs/server-setup.md`.
- Перед изменением короткого локального runbook читать `docs/local-server-run.md`.
- Перед изменением сетевого запуска backend/БД на другом ПК, remote API URL, firewall/CORS-инструкций или deployment-подхода читать `docs/remote-server-pc-setup.md`.
- Перед изменением интерфейса, CSS, визуального языка или layout читать `docs/visual-style.md`.
- Считать `implementation_plan.md` актуальным планом реализации и обновлять его при изменении поведения, стека, архитектуры или очередности работ.
- Считать `docs/data-architecture.md` простой технической картой данных проекта и обновлять её при изменении типов аккаунтов, прав доступа, источников данных, правил хранения, серверных контрактов или способов изменения данных.
- Считать `docs/server-setup.md` инструкцией по текущей backend-части и обновлять её при изменении `server/`, `docker-compose.yml`, env-переменных, миграций, API endpoints или команд запуска.
- Считать `docs/local-server-run.md` короткой инструкцией запуска уже настроенного локального сервера и не смешивать её с подробной настройкой.
- Считать `docs/remote-server-pc-setup.md` инструкцией по запуску backend/БД на отдельном ПК и обновлять её при изменении remote-подключения, CORS, портов, firewall-рекомендаций или deployment-подхода.
- Считать `docs/visual-style.md` каноническим визуальным направлением проекта и обновлять его только при осознанной смене дизайн-системы.

## Долговременные знания проекта

- Если в ходе запроса выясняется важное правило, которое должно влиять на будущие задачи, закреплять его в `AGENTS.md`.
- В `AGENTS.md` добавлять только устойчивые инструкции уровня подхода, безопасности, архитектурных границ, проверки и работы с данными.
- Не превращать `AGENTS.md` в журнал задач: детали реализации, временные решения и продуктовые очереди держать в `implementation_plan.md` или профильной документации.
- После любого изменения поведения, архитектуры, UI, контрактов или документации обновлять `implementation_plan.md`.
- В `implementation_plan.md` отмечать выполненные пункты, текущий активный пункт и ближайший следующий шаг.
- Не отмечать placeholder UI как готовую backend-функциональность: визуальный каркас, server boundary и реальные серверные данные должны быть явно разделены.

## Текущий стек

- Vite
- React
- TypeScript
- CSS без UI-фреймворка
- Backend workspace `server/` на Node.js + TypeScript
- Постоянная БД: MariaDB/MySQL

## Рабочий процесс

- Сначала изучать существующую структуру проекта, соседние файлы, зависимости и уже принятые паттерны.
- Считать frontend и backend единым проектным контуром в одном репозитории: при изменении серверных контрактов, API, CORS/env, deployment-подключения, форм, сохранения данных или access/profile сразу обновлять все связанные части — `src/`, `server/`, типы контрактов, тесты и релевантные docs.
- Учитывать серверный сценарий обновления через Git: всё, что нужно для работы backend/frontend после `git pull`, должно быть внесено в tracked-файлы проекта; ручные действия на сервере допустимы только для непубличных `.env`, секретов, установки зависимостей, миграций и перезапуска сервисов, и должны быть явно описаны.
- `server/.env` находится в рабочем проекте и может правиться напрямую для runtime-настроек backend; секретные значения не выводить в ответы, логи или документацию.
- Расширять текущую структуру `src/App.tsx`, `src/content.ts`, `src/styles.css`, не создавать параллельные системы без необходимости.
- Типы серверных контрактов, аккаунтов, доступов, статусов, показателей и оргструктур держать типизированными и централизованными.
- Не добавлять новые зависимости, backend, auth, chart-библиотеки, SDK интеграций или state manager без реальной задачи из плана.
- Backend-код держать в `server/`; не смешивать серверную валидацию, БД-запросы и API handlers с React/Vite frontend.
- Сохранять пользовательские изменения и не откатывать чужую работу.

## Данные и визуал

- Чётко отделять данные от визуального слоя.
- Все бизнес-данные, пользовательские данные, права доступа, аналитика, статусы, история, подтверждения и интеграционные payload должны храниться и проверяться на сервере.
- Клиент не должен быть источником истины: он запрашивает данные у сервера, отображает полученный ответ и отправляет пользовательские действия обратно на сервер.
- Во frontend допустимы типы контрактов, UI-состояния, состояние загрузки/ошибки и статический текст интерфейса; реальные бизнес-данные и демо-фикстуры с приватным смыслом не добавлять.
- Клиентские service-модули допустимы только как границы запросов к серверу; не добавлять в них fallback-данные, mock-ответы, локальные права или бизнес-фикстуры.
- Клиентские проверки доступа считать только UX-подсказкой. Защищённые действия всегда должны проверяться сервером.
- Локальный Vite middleware в `vite.config.ts` считать только dev/preview server boundary, а не production backend и не БД; не обещать сохранность данных, пока не подключено постоянное серверное хранилище.
- Для действий, которые должны сохраняться в удалённой БД, использовать remote API boundary; исключение допустимо только для явно тестового fallback: временная dev-сессия в `sessionStorage` для входа в интерфейс и диспетчерские отправки в `localStorage`, когда сервер или `/api` хостинга не настроены или недоступны. Такой режим должен быть визуально помечен как локальный тестовый, не считаться production-хранением/auth и сопровождаться тестами/документацией.
- Постоянное хранение реализовывать через backend workspace `server/` и MariaDB/MySQL; изменения схемы делать через миграции, а не ручные ad-hoc SQL правки в коде обработчиков.
- В remote/server-PC/VPS режиме открывать в сеть только backend API; MariaDB/MySQL держать доступной только локально для backend или по явно разрешённому IP, не публиковать порт `3306` наружу без отдельного осознанного решения.
- Визуальный стиль проекта: Industrial Finance Operations — светлый stone-gray рабочий интерфейс с blue-gray навигацией, compact density, server/status strip, очередями, таблицами и строгими статусными цветами.

## Проверки

После изменений запускать релевантные проверки:

```bash
npm test
npm run typecheck
npm run build
```

Для локальной проверки интерфейса:

```bash
npm run dev -- --host 127.0.0.1
```
