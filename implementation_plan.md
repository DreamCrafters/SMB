# План реализации SMB Monitor

## Цель продукта

SMB Monitor — платформа для мониторинга ключевых показателей бизнеса и их динамики во времени. Один пользовательский аккаунт может управлять несколькими бизнес-аккаунтами, а один бизнес должен поддерживать работу до 50 одновременных пользователей.

Платформа должна разделять четыре типа аккаунта: администратора с максимальными правами, владельца бизнеса с просмотром всей статистики своего бизнеса, работника с доступом только к отдельным рабочим функциям и диспетчера с доступом только к отправке диспетчерской формы. Система также должна поддерживать две модели оргструктуры, проводить данные через подтверждение перед использованием в аналитике, сохранять историю исправлений и готовиться к интеграциям с 1C, Excel, Google Sheets, Email и MAX.

## Технические правила

- До отдельной задачи не добавлять backend, auth provider, state manager, chart library, UI framework, SDK интеграций и новые зависимости.
- Держать типы аккаунтов, доступы, статусы, периоды, показатели и оргструктуру типизированными и централизованными.
- UI должен быть тонким: отображение, ввод и вызов прикладной логики.
- Бизнес-правила, проверки, переходы статусов, права доступа и подготовку данных держать отдельно от JSX и CSS.
- Любые формы, импорты, подтверждения, исправления, доступы, аудит и интеграции считать security-sensitive и покрывать проверками.
- Не использовать реальные приватные данные в тестовом контенте интерфейса.
- Все реальные бизнес-данные, пользовательские данные, права, аналитика, подтверждения и история должны храниться и проверяться на сервере.
- Клиент хранит только типы контрактов, UI-состояние и статический текст интерфейса; итоговые данные и разрешения получает по запросу к серверу.
- В remote/server-PC режиме открывать в сеть только backend API; PostgreSQL оставлять локальным для backend и не публиковать порт `5432` наружу без отдельного решения.
- `docs/data-architecture.md` обновлять вместе с изменениями модели данных, аккаунтов, доступов и серверных контрактов.
- `docs/local-server-run.md` держать коротким runbook запуска уже настроенного локального сервера без подробной настройки.
- `docs/remote-server-pc-setup.md` обновлять вместе с изменениями remote API URL, CORS, портов, firewall-рекомендаций и запуска backend/БД на другом ПК.
- `docs/visual-style.md` считать источником визуального языка: light stone-gray, ink text, blue-gray navigation, emerald/amber/brick статусы, compact density, left rail и server/status strip.

## Последовательность реализации

### 1. Базовая серверная модель аккаунтов и данных

Сначала зафиксировать минимальные типы серверных контрактов: бизнес-аккаунт, пользователь, тип аккаунта, область доступа, выданные сервером права, отдел, оргструктура, показатель, период, статус ввода данных, статус подтверждения, событие аудита и канал интеграции.

Реализованная часть шага: четыре типа аккаунта `admin`, `business_owner`, `worker`, `dispatcher`; серверный ответ access/profile; контракты business account, user profile, department, org structure mode, data entry status, confirmation status, dispatcher form definitions, dispatcher submissions, dispatcher feed summary и audit event summary. Клиент не хранит локальную матрицу прав как источник истины и не содержит бизнес-фикстуры.

Результат: типизированная основа, на которую можно опирать UI и backend-контракты без смешивания данных с визуалом. Клиентская boundary для `access/profile` уже существует, а первый backend-контур реализован для dispatcher submissions; остальные backend-домены ещё впереди.

Проверки: typecheck; unit-тесты добавлять тогда, когда появятся функции правил доступа, валидации или переходов статусов.

### 2. Каркас приложения и навигации

Собрать первый реальный экран без старого визуального прототипа: запрос текущего access/profile с сервера, отображение типа аккаунта, переключение доступных разделов и пустые рабочие области там, где сервер ещё не отдаёт данные.

Первый shell уже собран как визуальная основа без production backend: business/account значения запрашиваются через `access/profile` boundary. В локальном Vite dev server endpoint отвечает профилем выбранной dev-сессии или `{"profile": null}`, поэтому стартовый экран показывает окно выбора типа аккаунта без клиентских фикстур. После входа пользователь может выйти из временного dev-аккаунта через левый rail и вернуться к выбору типа доступа; рабочие области не показывают декоративные placeholder-плашки вместо отсутствующих серверных данных.

Результат: пользователь видит структуру продукта, но без фальшивой аналитики и без приватных данных.

Проверки: typecheck, build, ручной запуск через Vite.

### 3. Матрица ролей и доступов

Описать capabilities для серверного профиля доступа: управление бизнес-аккаунтами, управление пользователями, управление интеграциями, просмотр аудита, просмотр логов, использование debug-функций, просмотр всей статистики бизнеса, просмотр статистики отдела, отправка форм, отправка диспетчерских форм, просмотр диспетчерской истории, просмотр уведомлений и просмотр собственных отправок.

Реализованная часть шага: capability-значения централизованы в `src/contracts/accounts.ts`, `access/profile` отклоняет неизвестные capability, локальный Vite server выдаёт dev-профили с разными capabilities для admin, business owner, worker и dispatcher, а admin-профиль показывает только реальные capabilities из server profile без заглушечных административных панелей.

Результат: интерфейс получает права от сервера и использует их только для отображения доступных действий; сервер остаётся единственным источником защиты.

Проверки: unit-тесты на позитивные, запрещённые и пограничные сценарии доступа.

### 4. Оргструктура

Реализовать две модели отображения: классическую и существующую/current. Переключение модели не должно терять контекст выбранного бизнеса, отдела, периода и аналитики.

Результат: владелец бизнеса и администратор могут менять представление структуры без изменения исходных данных.

Проверки: тесты преобразования/выбора структуры, typecheck, build.

### 5. Ввод данных отделами

Сделать поток ввода данных для работника: выбор доступной формы, отдела или рабочего контекста, периода, показателя, натурального объёма, денежного значения, комментария и источника.

Текущее состояние: визуальная форма показана для роли `worker`, а диспетчер сначала выбирает одну из 6 серверно описанных диспетчерских форм отдельными кнопками и может вернуться к выбору из анкеты. Диспетчерская отправка идёт через remote boundary `VITE_SMB_REMOTE_API_URL` в backend workspace `server/`, где сервер валидирует `formId`, разрешённые поля `payload`, даты, числа и select-значения, затем сохраняет запись в PostgreSQL. Production auth и server-side capability checks ещё не реализованы.

Результат: черновик записи создаётся в статусе ожидания подтверждения и ещё не попадает в аналитический источник.

Проверки: валидация пустых, некорректных, отрицательных и неожиданных значений; unit-тесты правил создания записи.

### 6. Подтверждение данных другими отделами

Добавить очередь подтверждений: какой отдел внёс данные, какой отдел должен подтвердить, текущий статус, комментарии, решение и timestamp.

Результат: только подтверждённые записи становятся источником аналитики.

Проверки: тесты переходов статусов, запрет самоподтверждения там, где оно недопустимо, проверка истории решений.

### 7. Исправления и история

Реализовать поток корректировки уже существующих данных: причина исправления, автор, дата, старое значение, новое значение, статус исправления и связь с исходной записью.

Результат: ошибки, перенос бумажных данных и последующие корректировки не затирают историю.

Проверки: unit-тесты неизменности аудита и корректных переходов статусов.

### 8. Аналитические периоды и KPI

Собрать расчётные правила для последнего дня, текущего неполного месяца, текущего неполного года и динамики по выбранному периоду. Покрыть области: работники, входящие ресурсы, выполненные действия, полученный результат, задачи и активность пользователей.

Результат: аналитика строится из подтверждённых данных и явно учитывает неполные периоды.

Проверки: unit-тесты расчётов, граничные даты, пустые периоды, неполный месяц и неполный год.

### 9. Интерфейс владельца бизнеса

Собрать dashboard summary, переключатель оргструктуры, вкладки отделов, вложенные аналитические разделы, диспетчерскую ленту и уведомления.

Реализованная часть шага: добавлена отдельная вкладка владельца `Диспетчерская`, которая запрашивает live-историю отправленных диспетчером регистраций с backend API. Левый rail владельца является единственным переключателем между `Обзор` и `Диспетчерская`; `Обзор` пока оставлен пустым, а `Диспетчерская` не показывает placeholder-плашки и выводит фильтры, быстрые счётчики и строки live feed, когда backend возвращает данные.

Результат: владелец бизнеса получает обзор своего бизнеса, динамику и проблемные зоны без доступа к платформенным административным возможностям.

Проверки: typecheck, build, ручная проверка основных сценариев.

### 10. Расширенный рабочий интерфейс

Собрать формы ввода и корректировки данных, доступную статистику по серверно выданным правам и напоминания о событиях.

Результат: работник или пользователь с расширенными рабочими правами может поддерживать данные своего направления и видеть только разрешённую аналитику.

Проверки: тесты валидации форм и прав доступа, typecheck, build.

### 11. Ограниченный интерфейс работника

Ограничить worker-flow разрешённым вводом данных и минимальной навигацией без административных и аналитических возможностей сверх доступа.

Результат: работник может выполнить только разрешённые действия.

Проверки: тесты запрета лишних capabilities.

### 12. Интерфейс администратора

Добавить управление бизнес-аккаунтами, пользователями, типами аккаунтов, должностями, отделами, базой аккаунтов и аналитической базой. Отдельно предусмотреть смену паролей при ротации сотрудников.

Результат: администратор может обслуживать структуру платформы без прямого вмешательства в аналитику вне разрешённых действий.

Проверки: тесты административных прав, запретов для неадминистраторов и валидации критичных действий.

### 13. Диспетчерская лента и уведомления

Собрать единый поток событий: новые записи, ожидание подтверждения, отклонения, исправления, напоминания, просрочки и интеграционные ошибки.

Реализованная часть шага: для диспетчерских отправок добавлен первый live-feed contract `GET /api/dispatcher/submissions` через remote API. Endpoint поддерживает фильтры `formId`, `dateFrom`, `dateTo` и возвращает `summary` по количеству регистраций. Это ещё не единая диспетчерская лента всех событий.

Результат: владелец бизнеса, администратор и ответственные работники видят разрешённые рабочие события в одном месте.

Проверки: тесты фильтрации событий по роли, бизнесу, отделу и статусу.

### 14. Контракты интеграций

Подготовить adapter-контракты для 1C, Excel, Google Sheets, Email и MAX без реальных секретов и SDK. Для Excel и Google Sheets отдельно описать настройку форматирования импорта.

Результат: интеграции имеют безопасные границы и могут добавляться по одной без переписывания доменной логики.

Проверки: тесты нормализации входных данных, ошибок импорта и отсутствия логирования приватных payload.

### 15. Backend и постоянное хранение

После стабилизации frontend-прототипа выбрать backend, схему хранения, auth, audit storage и API-контракты. Frontend-only проверки доступа не считать достаточной защитой.

Текущее состояние: backend workspace `server/` уже поддерживает dispatcher submissions через PostgreSQL, серверные определения 6 диспетчерских форм, хранение `payload` в JSONB, краткую `summary` и фильтрованный feed. Frontend подключается к нему через `VITE_SMB_REMOTE_API_URL`. Production auth, полноценная модель пользователей, audit trail и общая аналитическая БД ещё не реализованы.

Результат: появляется реальная доверенная граница для данных, ролей, подтверждений, истории и интеграций.

Проверки: тесты API-контрактов, авторизации, валидации входа, миграций и audit trail.

### 16. Финальная полировка MVP

Проверить производительность ключевых экранов, пустые состояния, ошибки, адаптивность, доступность, безопасность логов и отсутствие приватных тестовых данных.

Результат: MVP готов к демонстрации и дальнейшей интеграционной разработке.
