# Архитектура данных SMB Monitor

Этот документ описывает данные проекта простым языком: какие сущности есть, где они должны храниться, кто может их менять и какие правила важно не потерять при разработке. Это не документация по конкретному коду и не схема базы данных.

## Главный принцип

Сервер является единственным источником истины для бизнес-данных, пользовательских данных, прав доступа, аналитики, подтверждений, истории и интеграций.

Клиентская часть должна только:

- запрашивать данные у сервера;
- отображать полученный ответ;
- собирать пользовательский ввод;
- отправлять действие на сервер;
- показывать состояния загрузки, ошибки, пустого результата и запрета доступа.

Клиентская часть не должна хранить реальные бизнес-данные, вычислять окончательные права доступа или подменять серверные правила локальными фикстурами.

## Типы аккаунтов

### Администратор

Администратор имеет максимальные права внутри платформы.

Может:

- управлять бизнес-аккаунтами;
- управлять пользователями и их доступами;
- управлять настройками аналитической базы;
- управлять интеграциями;
- смотреть аудит и техническое состояние;
- выполнять сервисные действия, если они разрешены серверной политикой.

Администраторские действия считаются критичными и должны проверяться на сервере.

### Владелец бизнеса

Владелец бизнеса управляет конкретным бизнес-аккаунтом и может просматривать всю статистику по этому бизнесу.

Может:

- видеть сводную статистику бизнеса;
- видеть аналитику по отделам;
- видеть динамику по периодам;
- видеть уведомления и диспетчерскую ленту по бизнесу;
- смотреть данные в рамках своего бизнес-аккаунта.

Владелец бизнеса не получает автоматически платформенные права администратора.

### Работник

Работник имеет ограниченный доступ к отдельным функциям.

Может только то, что явно разрешено сервером, например:

- заполнять доступные формы;
- отправлять данные на проверку или подтверждение;
- получать уведомления;
- видеть статус своих отправок;
- выполнять ограниченные рабочие действия по своему отделу или должности.

Работник не должен видеть глобальную статистику бизнеса, административные настройки, чужие данные и техническую информацию, если сервер не выдал отдельное разрешение.

## Права доступа

Права доступа выдаются сервером вместе с профилем текущей сессии или отдельным ответом access/profile. Клиент может использовать эти права для отображения или скрытия элементов интерфейса, но не может считать это защитой.

Любое защищённое действие должно повторно проверяться сервером:

- создание и изменение пользователей;
- просмотр статистики;
- отправка форм;
- подтверждение данных;
- корректировка записей;
- импорт данных;
- изменение интеграций;
- просмотр аудита.

Если клиент не получил право в ответе сервера, интерфейс должен считать действие недоступным.

## Основные группы данных

### Аккаунты и пользователи

Хранятся на сервере:

- пользовательские аккаунты;
- бизнес-аккаунты;
- тип аккаунта;
- связь пользователя с бизнесом;
- должность, отдел и рабочий контекст;
- серверно выданные права;
- состояние сессии.

### Организационная структура

Хранятся на сервере:

- отделы;
- должности;
- связи подчинения;
- классическая модель структуры;
- текущая фактическая модель структуры;
- правила переключения отображения.

Клиент показывает выбранное представление, но не меняет исходную структуру без серверного действия.

### Формы и рабочие записи

Хранятся на сервере:

- заполненные формы;
- черновики, если продукт разрешает серверные черновики;
- отправленные записи;
- статусы проверки и подтверждения;
- автор, время, отдел и источник записи;
- комментарии и причины отклонений.

### Аналитика

Хранятся и рассчитываются на сервере или доверенном backend-слое:

- показатели бизнеса;
- статистика по отделам;
- динамика по периодам;
- текущий день;
- неполный текущий месяц;
- неполный текущий год;
- агрегаты для дашбордов.

Клиент получает уже разрешённый для текущего аккаунта результат.

### История и аудит

Хранятся на сервере:

- просмотры;
- изменения;
- подтверждения;
- исправления;
- автор действия;
- timestamp;
- причина изменения;
- старое и новое значение там, где это допустимо политикой безопасности.

Аудит нельзя заменять клиентскими логами.

### Интеграции

Хранятся и выполняются на сервере:

- подключение 1C;
- импорт Excel;
- импорт Google Sheets;
- Email-уведомления;
- MAX-уведомления;
- настройки форматов;
- ошибки интеграций.

Секреты, токены и приватные payload не должны попадать в frontend.

## Где менять данные и правила

- Типы контрактов между клиентом и сервером: `src/contracts/`.
- Клиентские функции запроса серверных данных: `src/services/`.
- Простое описание data architecture: `docs/data-architecture.md`.
- Последовательность реализации и статус крупных этапов: `implementation_plan.md`.
- Инструкции для будущих AI-агентов и устойчивые правила работы: `AGENTS.md`.
- В будущем серверные правила, права, валидация, расчёты и хранение должны жить в backend/domain слоях, а не в React-компонентах.

## Клиентская граница access/profile

`src/services/accessProfile.ts` — первая клиентская boundary-функция для серверных данных. Она:

- запрашивает `GET /api/access/profile`;
- отправляет cookies через `credentials: "include"`;
- принимает только JSON-ответ;
- проверяет минимальную форму `ServerUserProfile`;
- возвращает состояния `ready`, `empty` или `error`;
- не содержит локальных пользователей, бизнесов, отделов, прав или fallback-фикстур.

React-интерфейс может хранить результат запроса только как UI-состояние для отображения. Этот результат не становится источником истины: защищённые действия всё равно должны проверяться сервером.

Для локальной разработки `vite.config.ts` поднимает серверный endpoint `GET /api/access/profile` и возвращает `{"profile": null}`. Это нужно, чтобы клиент проверял реальный HTTP boundary и показывал empty state. Этот endpoint не является production backend, не содержит реальных данных и не заменяет будущую серверную авторизацию.

## Текущие группы серверных контрактов

Контракты в `src/contracts/` описывают только форму серверных ответов и действий. Они не должны создавать локальные бизнес-данные или демо-фикстуры.

### Аккаунты и доступ

`src/contracts/accounts.ts` описывает:

- тип аккаунта;
- серверно выданные capabilities;
- область действия доступа;
- ответ `access/profile`;
- ошибки получения доступа.

### Бизнес, пользователь и оргструктура

`src/contracts/organization.ts` описывает:

- ссылку на бизнес-аккаунт;
- профиль пользователя, полученный с сервера;
- ссылку на отдел;
- режим оргструктуры: классический или текущий.

Эти данные можно менять только серверным действием. Клиент может выбрать отображение или отправить запрос на изменение, но не меняет исходные данные локально.

### Формы, отправки и подтверждения

`src/contracts/workflow.ts` описывает статусы:

- ввода данных;
- отправки данных;
- подтверждения;
- отклонения;
- запроса исправления.

Переходы между статусами должны проверяться сервером. Клиент может показывать статус и отправлять действие пользователя, но не утверждает итоговый статус сам.

### Аудит

`src/contracts/audit.ts` описывает краткую запись события аудита:

- кто выполнил действие;
- какое действие было выполнено;
- к какому объекту оно относится;
- когда это произошло;
- причина, если она есть и разрешена политикой безопасности.

Аудит хранится на сервере и не заменяется клиентскими логами.

## Текущий контракт первого этапа

Первый реализуемый слой фиксирует три типа аккаунта:

- `admin`;
- `business_owner`;
- `worker`.

Также зафиксированы первые серверные контракты:

- `BusinessAccountRef`;
- `ServerUserProfile`;
- `DepartmentRef`;
- `OrganizationStructureMode`;
- `DataEntryStatus`;
- `ConfirmationStatus`;
- `AuditEventSummary`.

Frontend хранит только TypeScript-контракты серверных ответов и временное UI-состояние запроса `access/profile`. Список реальных пользователей, бизнесов, отделов, статистики и выданных прав должен приходить с сервера.
